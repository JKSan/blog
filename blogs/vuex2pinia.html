把几个模块升级到pinia后，发现并总结了一些问题及其解决方法，可以提高升级过程的效率。 这里采用ts后缀文件，但其实并未真的全面应用ts。

一、仍然采用拆分多文件的方式，pinia store结构如下：

——moduleName
————state.ts
————getters.ts
————actions.ts
————model.ts
————index.ts

其中，原state、model不用动。

二、index.ts 统一这样写就行，其中ModuleName改为相应的模块名称。

import { defineStore } from 'pinia'
import state from './state'
import getters from './getters'
import actions from './actions'

export const useModuleNameStore = defineStore('moduleName', {
    state: () => state,

    getters: { ...getters },

    actions: { ...actions }
})

三、getters.ts 其中，原vuex第一个参数与pinia相同，都是state，而后面的getters、rootState、rootGetters在pinia中都没有。 对于vuex的getters参数，可以使用(this as any)替换，例如：

// getters.ts
export default {
    moduleNamesFiltered: state => {
        return state.moduleNames
    },
    getAllList () {
        return () => {
            return (this as any).moduleNamesFiltered
        }
    }
}

对于vuex的rootState和rootGetters参数，可分为两种情况。 如果目标模块已经迁移到pinia，则直接把目标模块引入即可；

import { useModuleNameStore } from '@/stores/moduleName'
const moduleNameStore = useModuleNameStore();
moduleNameStore.xxx

另一种方式，是整体把该getter方法，移到components中处理（在components节说明）。

四、原mutations 整体移入actions，对于addModuleName、deleteModuleName等重名问题，可以调整为addStateModuleName的形式。

五、actions.ts actions中可以引用index.ts中的useModuleNameStore，用于调用其自身的state和getters。 此外，对于rootState和rootGetters，与getters.ts的处理类似。 对于commit和dispatch，如果是调用actions内部方法，则使用(this as any).xxx；如果是其它模块方法，且其它模块已迁移pinia，则直接引入该模块；否则移入components处理。
